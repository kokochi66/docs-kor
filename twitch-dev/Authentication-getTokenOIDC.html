토큰 얻기 : OIDC 원본 사이트

시도

OIDC의 처리 절차에는 다음 두가지가 있습니다.

  • OIDC 암시적 코드 흐름을 통해서 ID 토큰과 부가적인 사용자 엑세스 토큰을 가져옵니다.
  • OIDC 인증 코드 흐름을 통해서 ID 토큰과 부가적인 사용자 엑세스 토큰을 가져옵니다.

자세한 다양한 타입과 절차에 대한 설명은 앱 과 인증 가이드를 확인하세요

OIDC 암시적 코드 흐름

1) 여러분이 인증하길 원하는 사용자의 정보를 등록된 리다이렉트 URI로 보냅니다. 인증 페이지에서는 사용자에게 트위치에 가입하거나, 로그인 하도록 요청하며, 사용자가 여러분의 어플리케이션의 시스템을 인증할 것인지 선택할 수 있게 도와줍니다.

1
GET https://id.twitch.tv/oauth2/authorize ?client_id=<your client ID> &redirect_uri=<your registered redirect URI> &response_type=<type> &scope=<space-separated list of scopes> &claims=<JSON object specifying requested claims>
cs

요청에는 몇가지 필수 파라미터와 옵션 파라미터가 존재합니다.

요구 파라미터 타입 설명
client_id string

여러분의 클라이언트 ID 입니다.
redirect_uri URI

여러분의 등록된 리다이렉트 URI입니다. 이는 등록단계에서 등록된 것과 정확히 일치해야합니다.
response_type string

두가지 값이 존재합니다. 하나는 token id_token으로, 엑세스 토큰과 ID 토큰을 반환합니다. 사용자에게는 오직 첫번쨰 요청에 대한 인증 확인 메세지만 표시됩니다. 이 방식은 token+id_token 이나 token%20id_token 형태의 URL로 인코딩 되어야 합니다. 다른 하나는 id_token 으로, 오직 ID 토큰만이 반환됩니다. 예를 들어, 유효한 엑세스 토큰을 가진 클라이언트가 인증된 사용자에 대한 추가 정보만이 필요한 경우에, 이 옵션을 선택할 수 있습니다.
scope string

범위(scope) 리스트를 공백으로 구분하여 입력합니다. 이 값은 openid 범위를 반드시 포함해야 합니다.
옵션 파라미터 타입 설명
claims string

UserInfo 엔드포인트의 응답 객체나 반환받은 ID 토큰에 대해 지정된 클레임(claims)을 문자열로 변환한 JSON 객체입니다.
nonce string

CSRF 공격을 피하기 위한 OIDC 의 불투명한 값입니다. (특히 이는 리플레이 공격(replay attacks)을 피하기 위하여, ID 토큰을 포함한 클라이언트 인증 세션과 연관되어 사용됩니다.) 이 지정한 값은 응답 객체로 다시 반환됩니다. 우리는 이 파라미터를 사용하는 것을 강력하게 추천합니다.
state string

여러분의 어플리케이션에 의해 생성된 여러분의 고유의 토큰입니다. 이는 CSRF 공격을 피하기 위한 OIDC의 불투명한 값입니다. 이 값은 응답 객체에서 동일하게 반환되며, 우리는 이 파라미터를 사용하는 것을 강하게 추천합니다.

다음은 viewing_activity_read 범위를 사용해서, 엑세스 요청을 보내고, 사용자를 http://localhost로 반환시키는 예제입니다.

1
GET 'https://id.twitch.tv/oauth2/authorize?response_type=token+id_token&client_id=uo6dggojyb8d6soh92zknwmi5ej1q2&redirect_uri=http://localhost&scope=viewing_activity_read+openid&state=c3ab8aa609ea11e793ae92361f002671&claims={"id_token":{"email_verified":null}}'
cs

2) 사용자가 여러분의 어플리케이션을 승인하면, 사용자의 정보가 여러분의 리다이렉트 URI로 ID토큰과 부가적으로는 엑세스 토큰과 함께 보내집니다.

1
https://<your registered redirect URI>#id_token=<an id token>&access_token=<an access token>
cs

JWT의 페이로드(payload)는 OIDC ID 토큰과 몇가지 여러분이 요청한 추가적인 클레임 값을 포함한 기본 클레임 객체를 반환합니다.

  • iss : 트위치의 토큰 발행자(issuer)
  • sub : 대상체(Subject) 또는 최종 사용자의 식별자
  • aud : 시청자 혹은 토큰에 의해 지정받은 OAuth 2.0 클라이언트 수신자
  • exp : 만기 기한(JWT ID 토큰의 만기 기한을 확인해야 합니다. 이것은 갱신되지 않습니다.)
  • iat - 배포 시간(Issuance time)
  • nonce - 지정한 요청에 대한 부가적인 값

다음 예제에서 ID 토큰은 요청에 의해 추가되는 email_verified 클레임 객체가 추가됩니다.

ID와 엑세스 토큰은 쿼리 스트링에 들어있지 않고, URL의 일부에 들어있습니다. 따라서 HTTP 요청으로 서버에 전달되지 않습니다. 이 객체는 document.location.hash를 이용한 자바 스크립트에서 접근할 수 있습니다.

응답객체는 URL 내부에 상태 파라미터를 포함하고, 여러분이 요청에 포함되어 있다면, ID 토큰의 페이로드(payload)에 nonce 파라미터가 포함됩니다.

3) ID 토큰을 식별합니다. 보안 검사는 토큰에 대한 여러가지 보안 공격을 대비하기 위해서 매우 중요한 작업입니다.

OIDC 인증 코드 흐름

1) 여러분이 인증하길 원하는 사용자의 정보를 등록된 리다이렉트 URI로 보냅니다. 인증 페이지에서는 사용자에게 트위치에 가입하거나, 로그인 하도록 요청하며, 사용자가 여러분의 어플리케이션의 시스템을 인증할 것인지 선택할 수 있게 도와줍니다.

1
GET https://id.twitch.tv/oauth2/authorize ?client_id=<your client ID> &redirect_uri=<your registered redirect URI> &response_type=code &scope=<space-separated list of scopes> &claims=<JSON object specifying requested claims>
cs

요청에는 몇가지 필수 파라미터와 옵션 파라미터가 존재합니다.

요구 파라미터 타입 설명
client_id string

여러분의 클라이언트 ID 입니다.
redirect_uri URI

여러분의 등록된 리다이렉트 URI입니다. 이는 등록단계에서 등록된 것과 정확히 일치해야합니다.
response_type string

두가지 값이 존재합니다. 하나는 token id_token으로, 엑세스 토큰과 ID 토큰을 반환합니다. 사용자에게는 오직 첫번쨰 요청에 대한 인증 확인 메세지만 표시됩니다. 이 방식은 token+id_token 이나 token%20id_token 형태의 URL로 인코딩 되어야 합니다. 다른 하나는 id_token 으로, 오직 ID 토큰만이 반환됩니다. 예를 들어, 유효한 엑세스 토큰을 가진 클라이언트가 인증된 사용자에 대한 추가 정보만이 필요한 경우에, 이 옵션을 선택할 수 있습니다.
scope string

범위(scope) 리스트를 공백으로 구분하여 입력합니다. 이 값은 openid 범위를 반드시 포함해야 합니다.
옵션 파라미터 타입 설명
claims string

UserInfo 엔드포인트의 응답 객체나 반환받은 ID 토큰에 대해 지정된 클레임(claims)을 문자열로 변환한 JSON 객체입니다.

여러분의 어플리케이션에 의해 생성된 여러분의 고유의 토큰입니다. 이는 CSRF 공격을 피하기 위한 OIDC의 불투명한 값입니다. 이 값은 응답 객체에서 동일하게 반환되며, 우리는 이 파라미터를 사용하는 것을 강하게 추천합니다.

1
bash GET 'https://id.twitch.tv/oauth2/authorize?response_type=code&client_id=uo6dggojyb8d6soh92zknwmi5ej1q2&redirect_uri=http://localhost&scope=viewing_activity_read+openid&state=c3ab8aa609ea11e793ae92361f002671&claims={"id_token":{"email_verified":null}}'
cs

2) 사용자가 여러분의 어플리케이션을 승인하면, 사용자의 정보가 여러분의 리다이렉트 URI로 ID토큰과 부가적으로는 엑세스 토큰과 함께 보내집니다.

1
https://<your registered redirect URI>/?code=<authorization code>

OAuth 2.0 인증 코드는 랜덤으로 생성되는 30개의 문자열로 이루어져 있습니다. 이는 다음단계에서 엑세스 및 ID 토큰과 교환하여 엔드포인트에 요청하기 위한 토큰으로 사용됩니다.

응답객체는 URL 내부에 상태 파라미터를 포함하고, 여러분이 요청에 포함되어 있다면, ID 토큰의 페이로드(payload)에 nonce 파라미터가 포함됩니다.

cs

3) 여러분의 서버에서, 다음 요청을 이용해서 엑세스 토큰과 ID 토큰을 받을 수 있습니다.

1
json POST https://id.twitch.tv/oauth2/token ?client_id=<your client ID> &client_secret=<your client secret> &code=<authorization code received above> &grant_type=authorization_code &redirect_uri=<your registered redirect URI>
cs

cs

  • exp : 만기 기한(JWT ID 토큰의 만기 기한을 확인해야 합니다. 이것은 갱신되지 않습니다.)
  • iat - 배포 시간(Issuance time)
  • nonce - 지정한 요청에 대한 부가적인 값

클레임(Claims)

클레임은 여러분의 어플리케이션에서 여러분이 얻길 원하는 사용자 인증과 관련된 데이터를 설명합니다. 여러분의 OIDC 요청의 claims 파라미터를 통해서 여러분들은 클레임을 포함 할 것인지, 어느정도 포함할 것인지를 설정할 수 있습니다.

클레임은 인증 요청에서 claims 파라미터를 전달함으로써, 요청할 수 있습니다. 이렇게 요청된 클레임은 ID 토큰이나 UserInfo 엔드포인트를 통해서 검색할 수 있습니다. 각 클레임 값을 반환 받으려면, 요청에서 claims 파라미터의 포맷을 어떻게 하느냐에 따라 결정할 수 있습니다.

claims 파라미터는 id_token과 userinfo 필드를 가진 JSON 객체 입니다. 여러분은 이러한 필드 값을 설정할 수 있으며, 필드값의 필수여부나, 고유성 여부에 대한 요구사항은 존재하지 않습니다. 예를들어서, 사용자의 이메일과 이메일 인증 상태를 ID 토큰에 포함하고, Userinfo 엔드포인트를 이용한 프로필 이미지를 받기 위해서, claims 파라미터는 다음과 같이 설정합니다.

1
{ "id_token": { "email": null, "email_verified": null }, "userinfo": { "picture": null } }
cs

즉, 요청에서 claims 파라미터는 다음과 같이 지정하게 됩니다.

1
claims={"id_token":{"email":null,"email_verified":null},"userinfo":{"picture":null}}
cs

위에서 email과 email_verified 클레임은 반환되는 ID 토큰에 포함되어 나타납니다. picture 클레임은 여러분이 불러들인 UserInfo 엔드포인트에 인증 요청에서 반환 되었던 OAuth 토큰을 사용해서 요청을 보낼 시에, 반환됩니다. 이는, 여러분이 암시적, 혹은 인증 코드 흐름을 사용해서 OAuth 토큰을 요청했다고 가정했을 때 입니다.

여러분은 다음과 같은 클레임을 요청에 넣을 수 있습니다.

클레임 데이터
email 권한을 부여하는 사용자의 이메일 주소 입니다.
email_verified 권한을 부여하는 사용자의 이메일 인증 상태입니다.
picture 권한을 부여하는 사용자의 프로필 이미지 URL 입니다.
preferred_username 권한을 부여하는 사용자의 표시되는 이름입니다.
updated_at 권한을 부여하는 사용자의 프로필의 최신 업데이트 날짜입니다.

참고 : email과 email_verified 클레임은 승인 요 에서 포함되어야 하는 user:read:email 범위에 의해서 연결 될 것입니다.

다음 클레임은 ID 토큰과 UserInfo 엔드포인트 의 디폴트 값에 포함됩니다.

디폴트 클레임 데이터
aud Audience : 사용자의 승인 요청을 보내는 어플리케이션의 클라이언트 ID를 의미합니다.
azp Authorized pary : 승인되고있는 어플리케이션의 클라이언트 ID를 의미합니다. 거의 aud와 동일합니다.
exp 토큰의 만기기한 입니다. 이는 UNIX/Epoch 포맷으로 이루어져 있습니다.
iat 토큰이 발행된 시간입니다. 이는 UNIX/Epoch 포맷으로 이루어져 있습니다.
iss 토큰의 발행자 데이터 입니다.
sub 인증하는 사용자의 사용자 ID 데이터 입니다.

UserInfo 엔드포인트

UserInfo 엔드포인트는 ID 토큰 외부의 권한 요청에서 원래 제공되어있던 클레임을 가져오는 역할을 합니다. 이 엔드포인트는 권한을 부여하는 사용자을 대신해서, OAuth 토큰을 받은 경우에 사용이 가능합니다. 여기서, 여러분의 인증 요청이 id_token 응답 타입만을 지정한다면, 이 엔드포인트를 호출 할 OAuth 토큰이 없게 될 것입니다.

1
GET 'https://id.twitch.tv/oauth2/userinfo' -H "Authorization: Bearer <OAuth token>"
cs

응답 객체는 JSON form으로 클레임을 요청합니다.

1
{ "aud": "<your client ID>", "exp": 1542051527, "iat": 1542050627, "iss": "https://id.twitch.tv/oauth2", "sub": "<authorizing user ID>", "azp": "<your client ID>", "picture": "<authorizing user profile image>" }
cs
